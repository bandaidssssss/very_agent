二、显存占用说明

由于hybrid_engine下强化学习分为多个阶段，不同阶段之间既有独立显存，也有部分tradeoff，因此需要对这部分进行说明。

1. 每一个时间步(step)的显存分为rollout_memory, log_prob_memory, ref_prob_memory, training_memory
 2. 其中rollout model一般常驻于显存当中，actor model一般常驻于显存（actor.param_offload=False)，ref model一般不常驻显存（ref.param_offload=True）
 3. rollout生成的样本，以平均截断长度来算，有(response_length/mean + prompt_length/mean)*data.train_batch_size*rollout.n样本，这是一个与data.train_batch_size线性相关的显存占用，且这些tokens常驻于当前step的显存当中。
 4. rollout_memory相对独立，仅与actor model的显存有tradeoff。vllm kv cache是独有的，这部分会通过开启free_cache_engine在rollout后被释放。
 5. log_prob_memory独立性最小，受到rollout model和actor model显存，以及rollout kv-cache的制约（rollout阶段显存通常还未完全释放）。因此log_prob阶段的log_prob_micro_batch_size_per_gpu是瓶颈，通常也是整个step的显存峰值。
6. ref_prob_memory相对独立，除了常驻于显存的模型和样本的tokens没有其他与之共享的显存部分。
7. training_memory相对独立，此阶段只有rollout model的显存有tradeoff，但是制约此显存的训练参数较多
8. 当调整影响rollout model和actor model副本显存参数的时候，要考虑各阶段显存的tradeoff。
三、专家经验

要求调整参数时满足以下约束关系

显存/吞吐相关

单步预测tokens数固定。rollout.n*data.train_batch_size*(data.max_prompt_length+data.max_response_length)，这也是为了保持样本的常驻显存大小固定。
迭代步数相同。data.train_batch_size/trainer.total_epochs 固定。
batch整除关系。data.train_batch_size*rollout.n  %  actor.ppo_mini_batch_size == 0, actor.ppo_mini_batch_size* % actor.ppo_micro_batch_size_per_gpu == 0。为了达到最高的训练MFU，可以考虑actor.ppo_micro_batch_size_per_gpu =3，data.train_batch_size为3的倍数这种可能性，因为2->4的对训练显存的增加过大。
如果要增大actor单卡模型权重（降低tensor_model_parallel_size和pipeline_model_parallel_size），需要相应减小rollout.gpu_memory_utilization以及和rollout显存相关的参数防分配rollout_memory分配不足导致的溢出。并适当降低rollout.log_prob_micro_batch_size_per_gpu以确保log_prob_memory不溢出。同时可能需要适当减小actor.ppo_micro_batch_size_per_gpu或者开启actor.override_transformer_config中的recompute相关参数，以防止train_memory溢出。反之即当actor的权重显存变小时，可以增大其他部分的显存开销。
增大ppo_micro_batch_size_per_gpu时会较大幅增加train_memory中的激活值，当需要满足整除关系导致actor.ppo_micro_batch_size_per_gpu需要成倍增加时如2->4, 4->8, train_memory很容易溢出，此时需要可能需要开启actor.override_transformer_config中的recompute相关参数防止溢出。
recompute_granularity为selective时，通过修改recompute_modules来选择重计算的单元。仅当recompute_granularity为full时，开启其他方式的重计算。
ref_memory相对独立。可以较大幅度降低ref.megatron.tensor_model_parallel_size以及提高ref.log_prob_micro_batch_size_per_gpu
当actor.param_offload为True时，rollout阶段显存可能存在大量浪费，占用率极低，但是训练显存依旧高。仅vllm开销过大吞吐严重不足成为主要瓶颈时开启此项
收敛速度相关：

actor.optim.lr对收敛速度影响过大，但是过大的lr会导致reward震荡无法收敛。根据日志输出的reward/mean轨迹调整lr，调整范围建议在1e-6~1e-5这个量级，单次步长在2e-6之内
kl_loss_coef限制模型不要远离SFT的模型，过大会限制模型的探索性，根据日志输出的reward/mean轨迹和。建议在1e-6~1e-5这个量级，单次调整步长在0.002以内
rollout.n通过影响优势函数（Advantage）的估计质量，显著决定了模型在训练中能否稳定地达到更高的奖励峰值。该值存在一个边际收益，当从一个较小值变化到一个中等值（如4->8），能显著减少训练噪声达到稳定训练的效果。但是rollout.n增加会带来额外的显存压力（当train_batch_size不变），线性增加单个epoch训练所需的时间。
data.max_prompt_length/max_response_length影响模型能力，能正面收敛。建议通过对数据集分布的事前分析，以及reward稳定后的平均response_len，平均response_len_clip_ratio，平均prompt_len等指标确定这两个值。增大这两个值会严重影响显存占用（不减小batch_size和rollout.n）或极大地增加训练时间。建议按照模型刚需划线，或保持在数据集中率90%以上的序列长度
